# Tarot Reflection App

A personal, AI-assisted tarot app for reading, reflection, and learning (not fortune-telling).
Angular 20 PWA + Supabase + Google Gemini, hosted on Vercel.

**Read first:** [`FEATURES.md`](FEATURES.md) (full spec) and
[`supabase/migrations/0001_tarot_init.sql`](supabase/migrations/0001_tarot_init.sql) (the schema —
source of truth for the DB).

## Hard rules (do not violate)

1. **Shared database — tables MUST stay separate.** This app piggybacks on the **life-assistant
   Supabase project** (ref `snlrpqamjzwoksmoxzir`). There is no separate DB.
   - **Every tarot table, type, function, and trigger is prefixed `tarot_`.** Generic names
     (`profiles`, `cards`, `readings`, `set_updated_at`, `handle_new_user`, …) would collide with
     life-assistant.
   - **Never add a trigger to `auth.users`** (shared across both apps). The `tarot_profiles` row is
     upserted client-side on first load — see [`src/app/core/profile.service.ts`](src/app/core/profile.service.ts).
   - Storage uses a dedicated **public** bucket `tarot-card-art`.

2. **RLS is per-user.** User content is owner-scoped via `auth.uid()`; shared/built-in content uses
   `is_public OR owner_id = auth.uid()`; reference data is read-only to authenticated users. Never
   ship `using (true)` policies for tarot tables.

3. **AI = Google Gemini (`gemini-2.5-flash`) via a Supabase Edge Function — never from the browser.**
   Reuse the `GEMINI_API_KEY` secret and the life-assistant `_shared/gemini.ts` pattern. Theme
   search and synthesis read interpretation **keywords**, not full prose, to keep calls cheap.

4. **Canonical card structure is locked to RWS-78** (strict enums + `tarot_card_shape` check).

5. **Multi-card spots are first-class** (`card_count` / `slot_index`, one shared position `meaning`,
   read together). Reference spread: the **Cross of Kells** (FEATURES §3.4).

## Conventions
- Angular standalone components + signals, `OnPush`. Feature dirs under `src/app/features/`, shared
  singletons under `src/app/core/`. Tailwind v4 tokens in `src/styles.css`.
- Migrations: `supabase/migrations/NNNN_*.sql`. Edge Functions: `supabase/functions/`.
- Reference/seed data is generated by `supabase/seed/build-seed.mjs` (`npm run seed:build`).
- **Cloud writes need Rory's explicit go-ahead** — do not `supabase db push` / apply seeds to the
  shared cloud DB without confirming (it's the live life-assistant database).
- Commit + push at the end of each task (Rory tests remotely).

## Schema note
The migration SQL is valid **PostgreSQL**. VS Code's default SQL linter parses T-SQL and flags
`create type ... as enum`, RLS policies, etc. as errors — false positives; ignore or switch the
editor SQL dialect to PostgreSQL.
